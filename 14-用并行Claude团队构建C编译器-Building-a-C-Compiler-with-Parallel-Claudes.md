# 用并行Claude团队构建C编译器 (Building a C compiler with a team of parallel Claudes)

> **原文链接**: https://www.anthropic.com/engineering/building-c-compiler  
> **发布日期**: 2026年2月5日  
> **作者**: Nicholas Carlini

---

## 摘要

我们让Opus 4.6使用智能体团队构建一个C编译器，然后（大部分）走开了。以下是它教给我们的关于自主软件开发未来的东西。

我一直在试验一种我们称之为"智能体团队"的新方法来监督语言模型。

通过智能体团队，多个Claude实例在共享代码库上并行工作，无需主动人工干预。这种方法极大地扩展了使用LLM智能体可以实现的目标范围。

为了对其进行压力测试，我让16个智能体用Rust从头开始编写一个C编译器，能够编译Linux内核。在近2,000个Claude Code会话和20,000美元的API成本中，智能体团队生成了一个100,000行的编译器，可以在x86、ARM和RISC-V上构建Linux 6.9。

该编译器本身是一个有趣的产物，但我这里关注的是我从为长运行自主智能体团队设计工具中学到的：如何编写测试让智能体在没有人工监督的情况下保持正轨，如何构建工作结构使多个智能体能够并行取得进展，以及这种方法在哪里达到其极限。

## 启用长运行的Claude

现有的智能体支架如Claude Code需要操作员在线并可用以共同工作。如果你要求解决一个长期而复杂的问题的解决方案，模型可能会解决其中的一部分，但最终它会停止并等待继续输入——一个问题、状态更新或澄清请求。

为了引发持续的自主进展，我构建了一个将Claude放入简单循环的工具（如果你见过Ralph-loop，这应该看起来很熟悉）。当它完成一个任务时，它立即接手下一个。

```bash
#!/bin/bash

while true; do
    COMMIT=$(git rev-parse --short=6 HEAD)
    LOGFILE="agent_logs/agent_${COMMIT}.log"

    claude --dangerously-skip-permissions \
           -p "$(cat AGENT_PROMPT.md)" \
           --model claude-opus-X-Y &> "$LOGFILE"
done
```

在智能体提示中，我告诉Claude要解决什么问题，并要求它通过将问题分解为小部分、跟踪它正在做什么、弄清楚接下来要做什么，并有效地继续直到完美来解决问题。

## 并行运行Claude

并行运行多个实例可以解决单智能体工具的两个弱点：

- 一个Claude Code会话一次只能做一件事。随着项目范围的扩大，并行调试多个问题要高效得多。
- 运行多个Claude智能体允许专业化。虽然一些智能体被分配解决手头的问题，但其他专门的智能体可以被调用来（例如）维护文档、关注代码质量，或解决专门的子任务。

我的并行Claude实现是简陋的。创建一个新的裸git仓库，对于每个智能体，启动一个Docker容器，将仓库挂载到`/upstream`。每个智能体克隆一个本地副本到`/workspace`，完成后，从其本地容器推送到upstream。

为了防止两个智能体试图同时解决同一个问题，工具使用一个简单的同步算法：

1. Claude通过在current_tasks/中写入文本文件来"锁定"一个任务
2. Claude处理任务，然后从upstream拉取，合并来自其他智能体的更改，推送其更改，并移除锁定
3. 无限智能体生成循环在一个新容器中生成一个新的Claude Code会话，循环重复

## 与Claude智能体团队编程的经验教训

该支架在循环中运行Claude，但只有Claude能够告诉如何取得进展时，该循环才有用。我的大部分精力都投入到设计Claude周围的环境——测试、环境、反馈——以便它能够在没有我的情况下定位自己。以下是我发现在协调多个Claude实例时最有帮助的方法。

### 编写极高质量的测试

Claude将自主工作来解决我给它的任何问题。因此，重要的是任务验证器几乎完美，否则Claude会解决错误的问题。改进测试工具需要找到高质量的编译器测试套件，为开源软件包编写验证器和构建脚本，并观察Claude犯的错误，然后在我识别这些失败模式时设计新测试。

### 站在Claude的角度思考

我必须不断提醒自己，我是为Claude而不是为我自己编写这个测试工具的，这意味着要重新思考我对测试应该如何传达结果的许多假设。

例如，每个智能体都被放入一个新鲜的容器中，没有上下文，会花费大量时间定位自己，特别是在大型项目上。在我们甚至到达测试之前，为了帮助Claude帮助自己，我包含了维护广泛的README和进度文件的指令，这些文件应该经常更新当前状态。

我还牢记语言模型有固有的局限性，在这种情况下，需要围绕这些局限性进行设计。这些包括：

- **上下文窗口污染**：测试工具不应该打印数千个无用的字节。最多，它应该只打印几行输出，并将所有重要信息记录到文件中，以便Claude在需要时可以找到。
- **时间盲目**：Claude无法分辨时间，如果放任不管，它会愉快地花费数小时运行测试而不是取得进展。工具很少打印增量进度（以避免污染上下文），并包含一个默认的`--fast`选项，运行1%或10%的随机样本。

### 让并行变得容易

当有许多不同的失败测试时，并行化是微不足道的：每个智能体选择一个不同的失败测试来处理。在测试套件达到99%通过率后，每个智能体致力于让不同的小型开源项目（如SQlite、Redis、libjpeg、MQuickJS、Lua）能够编译。

但当智能体开始编译Linux内核时，它们卡住了。与具有数百个独立测试的测试套件不同，编译Linux内核是一个巨大的任务。每个智能体都会遇到相同的bug，修复那个bug，然后覆盖彼此的更改。运行16个智能体没有帮助，因为每个都卡在解决相同的任务上。

解决方案是使用GCC作为在线已知良好的编译器oracle进行比较。我写了一个新的测试工具，随机使用GCC编译大部分内核，只使用Claude的C编译器编译剩余的文件。如果内核工作了，那么问题就不在Claude的文件子集中。如果它坏了，那么它可以通过使用GCC重新编译其中一些文件来进一步细化。这让每个智能体可以并行工作，修复不同文件中的不同bug，直到Claude的编译器最终能够编译所有文件。

### 多个智能体角色

并行还实现了专业化。LLM编写的代码经常重新实现现有功能，所以我让一个智能体负责合并它发现的任何重复代码。我让另一个负责提高编译器本身的性能，第三个我让它负责输出高效的编译代码。我让另一个智能体从Rust开发者的角度批评项目的设计，并对项目进行结构性更改以提高整体代码质量，另一个负责文档。

## 压力测试智能体团队的极限

这个项目被设计为一个能力基准测试。我有兴趣压力测试LLM今天刚好能够达到的极限，以帮助我们准备模型将来能够可靠实现的目标。

我一直在使用C编译器项目作为整个Claude 4模型系列的基准。与之前的项目一样，我从起草我想要的东西开始：一个从头开始的优化编译器，没有依赖，GCC兼容，能够编译Linux内核，并设计为支持多个后端。虽然我指定了设计的某些方面（例如它应该有SSA IR以启用多个优化通道），但我没有详细说明如何做到这一点。

之前的Opus 4模型几乎无法产生一个功能编译器。Opus 4.5是第一个跨越阈值允许它产生一个功能编译器的大型测试套件的模型，但它仍然无法编译任何真正的大型项目。我对Opus 4.6的目标是再次测试极限。

### 评估

在近2,000个Claude Code会话中，跨越两周，Opus 4.6消耗了20亿输入token并生成了1.4亿输出token，总成本略低于20,000美元。与即使是最昂贵的Claude Max计划相比，这是一个极其昂贵的项目。但总额只是我自己生产这个成本的一小部分——更不用说整个团队了。

这是一个干净的房间实现（Claude在开发过程中任何时候都没有互联网访问）；它只依赖于Rust标准库。100,000行的编译器可以在x86、ARM和RISC-V上构建可启动的Linux 6.9。它还可以编译QEMU、FFmpeg、SQlite、postgres、redis，并且在大多数编译器测试套件上有99%的通过率，包括GCC折磨测试套件。它还通过了开发者的终极试金石测试：它可以编译并运行Doom。

然而，编译器并非没有局限性。这些包括：

- 它缺少16位x86编译器，这是从实模式启动Linux所必需的。为此，它调用GCC（x86_32和x86_64编译器是它自己的）。
- 它没有自己的汇编器和链接器；这些是Claude开始自动化但仍然有些错误的最后部分。
- 编译器成功构建了许多项目，但不是所有项目。它还不是真正的编译器的直接替代品。
- 生成的代码不是很高效。即使启用了所有优化，它输出的代码也比GCC禁用所有优化时的效率低。
- Rust代码质量是合理的，但远不及专家Rust程序员可能产生的质量。

## 展望未来

每一代语言模型都开辟了与它们合作的新方式。早期模型在IDE中对制表符补全很有用。不久之后，模型可以从其文档字符串完成函数体。Claude Code的推出将智能体带入主流，并使开发者能够与Claude结对编程。但这些产品中的每一个都基于一个假设，即用户定义任务，LLM运行几秒钟或几分钟并返回答案，然后用户提供后续跟进。

智能体团队展示了自主实现整个复杂项目的可能性。这使我们作为这些工具的用户，能够对我们的目标变得更加雄心勃勃。

我们仍处于早期阶段，完全自主开发伴随着真正的风险。当人类在开发过程中与Claude坐在一起时，他们可以确保一致的质量并实时捕捉错误。对于自主系统，很容易看到测试通过并假设工作已经完成，而这种情况很少见。

因此，虽然这个实验让我兴奋，但它也让我感到不安。构建这个编译器是我最近最有趣的经历之一，但我没想到这在2026年初就能实现。语言模型和我们用于与它们交互的支架的快速进步为编写大量新代码打开了大门。我期望积极的应用会超过消极的，但我们正在进入一个新世界，这将需要新策略来安全导航。
